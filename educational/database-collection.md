## 1. 🚕 БАЗА ДАННЫХ ДЛЯ СЛУЖБЫ ТАКСИ

### Описание
База данных для управления заказами в службе такси. Система отслеживает клиентов, водителей, автомобили и поездки, а также позволяет оставлять отзывы о качестве обслуживания.

### Количество сущностей
5 таблиц

### Основные сущности
- **Клиент** — пользователи сервиса
- **Водитель** — водители такси
- **Автомобиль** — транспортные средства
- **Заказ** — поездки клиентов (структурный центр)
- **Отзыв** — оценки качества обслуживания

### Ключевые решения
- Заказ является структурным центром — связывает клиента, водителя и автомобиль
- Отзыв вынесен в отдельную таблицу для связи M:N между клиентом и водителем
- Добавлены статусы заказа (новый/в пути/завершён) для отслеживания состояния

### Связи
- 1:M: Клиент → Заказ (один клиент может сделать много заказов)
- 1:M: Водитель → Заказ (один водитель выполняет много заказов)
- 1:M: Автомобиль → Заказ (один автомобиль используется во многих заказах)
- M:N: Клиент ↔ Водитель (через таблицу Отзыв)

### Возможности развития системы

- Финансовый модуль: учёт способов оплаты (наличные, карта, онлайн), история транзакций, возвраты и чаевые водителям
- Система тарификации: разные тарифы (эконом, комфорт, бизнес), динамическое ценообразование в часы пик, коэффициенты района
- Геолокация: GPS-трекинг автомобилей, история маршрутов, расчёт оптимального пути, зоны покрытия
- Рейтинг водителей: автоматический расчёт рейтинга на основе отзывов, статистики поездок и жалоб клиентов


## 2. 🏫 БАЗА ДАННЫХ ДЛЯ ШКОЛЫ

### Описание
База данных для управления учебным процессом в школе. Система учитывает учеников, родителей, учителей, предметы, расписание и успеваемость учащихся.

### Количество сущностей
7-8 таблиц

### Основные сущности
- **Ученик** — учащиеся школы
- **Родитель** — законные представители
- **Учитель** — преподаватели
- **Предмет** — учебные дисциплины
- **Класс** — учебные группы
- **Расписание** — уроки
- **Оценка** — успеваемость учеников

### Ключевые решения
- Оценки вынесены в отдельную таблицу для отслеживания истории успеваемости
- Связь M:N между учеником и родителем (один ребёнок может иметь нескольких опекунов)
- Расписание связывает класс, учителя, предмет и время

### Связи
- 1:M: Ученик → Оценка
- M:N: Ученик ↔ Предмет (через таблицу Оценки)
- M:N: Ученик ↔ Родитель (через промежуточную таблицу)
- 1:M: Класс → Ученик

### Сложности
- Связь M:N между учеником и предметом через таблицу оценок
- Учёт нескольких родителей у одного ученика
- Разделение классного руководителя и учителей-предметников

### Возможности развития системы
- Модуль посещаемости: учёт пропусков, автоматические уведомления родителям, отчёты по классам для завуча
- Электронный журнал: домашние задания, темы уроков, прикрепление материалов, интеграция с электронным дневником
- Факультативы и кружки: отдельные расписания, запись учащихся, учёт загрузки преподавателей
- Экзаменационный модуль: расписание экзаменов, распределение аудиторий и наблюдателей, ведомости результатов


## 3. 🏨 БАЗА ДАННЫХ ДЛЯ ОТЕЛЯ

### Описание
База данных для управления бронированием номеров в отеле. Система отслеживает гостей, номера различных категорий, бронирования и посещения, а также работу персонала.

### Количество сущностей
6 таблиц

### Основные сущности
- **Гость** — постояльцы отеля
- **Номер** — гостиничные номера
- **Тип номера** — категории номеров (стандарт, люкс, семейный...)
- **Бронирование** — заказы номеров (план)
- **Посещение** — факты заселения (факт)
- **Сотрудник** — персонал отеля

### Ключевые решения
- Разделение бронирования (план) и посещения (факт) для учёта неявок
- Тип номера вынесен в справочник для избежания дублирования
- Добавлены статусы номера (свободен/занят/уборка/ремонт) для управления фондом

### Связи
- 1:M: Гость → Бронирование
- 1:M: Тип номера → Номер
- 1:M: Бронирование → Посещение
- 1:M: Сотрудник → Должность (справочник)

### Возможности развития системы
- Дополнительные услуги: учёт ресторана, спа, прачечной, мини-бара в номере с привязкой к счёту гостя
- Система лояльности: программы для постоянных гостей, накопительные бонусы, персональные скидки
- Управление фондом: учёт ремонта номеров, сезонные цены, блокировка номеров на период обслуживания
- Интеграции: подключение к системам онлайн-бронирования (Booking, Ostrovok)


## 4. 🏋️ БАЗА ДАННЫХ ДЛЯ ФИТНЕС-КЛУБА

### Описание
База данных для управления фитнес-клубом. Система учитывает клиентов, абонементы различных типов, посещения тренажёрного зала, групповые занятия с расписанием и индивидуальные тренировки с тренерами.

### Количество сущностей
10-12 таблиц

### Основные сущности
- **Клиент** — посетители клуба
- **Сотрудник** — тренеры и администраторы (включая специализацию тренеров)
- **Абонемент** — купленные клиентами пакеты услуг (структурный центр)
- **Тип абонемента** — виды абонементов (месяц, год, безлимит, по посещениям...)
- **Посещение клуба** — факты прихода в зал
- **Групповое занятие** — расписание групповых тренировок
- **Запись на групповое занятие** — бронирование мест
- **Индивидуальное занятие** — персональные тренировки
- **Тренажёрный зал** — помещения клуба
- **Справочник должностей** — роли сотрудников

### Ключевые решения
- Разделение абонемента (покупка права) и посещения (фактическое использование)
- Три промежуточные таблицы для связей M:N: Посещение клуба, Запись на групповое занятие, Индивидуальное занятие
- Абонемент является структурным центром — связывает клиента, тип абонемента и сотрудника
- Статус абонемента включает заморозку (активен/истёк/заморожен/аннулирован)
- Специализация тренеров хранится в таблице Сотрудник (не отдельная таблица)

### Связи
- 1:M: Клиент → Абонемент
- 1:M: Тип абонемента → Абонемент
- M:N: Клиент ↔ Тренажёрный зал (через Посещение клуба)
- M:N: Клиент ↔ Групповое занятие (через Запись на занятие)
- M:N: Клиент ↔ Тренер (через Индивидуальное занятие)

### Сложности
- Связь M:N между клиентом и групповым занятием через таблицу записи с контролем лимита мест
- Связь M:N между клиентом и тренером через индивидуальные занятия
- Учёт специализации тренеров (йога, кроссфит, плавание) в таблице Сотрудник
- Заморозка абонемента как статус, а не отдельное поле

### Возможности развития системы

 - Финансовый модуль: учёт оплат абонементов с историей транзакций, рассрочка платежа, интеграция с кассовой системой
 - Программа лояльности: карты постоянных клиентов, реферальная система "приведи друга", бонусы за регулярные посещения
 - Медицинский модуль: учёт противопоказаний, рекомендации врачей, фитнес-тестирования и замеры прогресса клиентов
 - Мобильное приложение: онлайн-запись на групповые занятия, push-уведомления, личный кабинет клиента


## 5. 💇 БАЗА ДАННЫХ ДЛЯ САЛОНА КРАСОТЫ

### Описание
База данных для салона красоты / SPA. Система управляет записью клиентов, оказанием услуг, учётом расходных материалов, расписанием работы мастеров и отзывами.

### Количество сущностей
12 таблиц

### Основные сущности
- **Клиент** — посетители салона
- **Сотрудник** — мастера и администраторы
- **Услуга** — предоставляемые услуги (стрижка, маникюр, окрашивание...)
- **Категория услуги** — виды услуг (волосы, ногти, косметика...)
- **Визит** — факты прихода клиентов (структурный центр)
- **Предоставление услуги (Состав визита)** — оказанные услуги в рамках визита
- **Материалы** — расходные материалы (краска, лак, крем...)
- **Расход материалов** — учёт использованных материалов
- **Оплата** — платежи клиентов
- **Отзыв** — оценки услуг и мастеров (две оценки: услуги и мастера)
- **Расписание** — график работы сотрудников
- **Справочник должностей**

### Ключевые решения
- Разделение визита (факт прихода) и предоставления услуги (конкретные оказанные услуги) — один визит может включать несколько услуг
- Материалы вынесены в отдельную таблицу с учётом остатков на складе и минимального остатка для заказа
- Расход материалов связан с визитом для отслеживания фактического использования
- Расписание вынесено в отдельную таблицу для управления загрузкой мастеров и поиска свободных окон
- Отзыв содержит две оценки: услуги и мастера
- Структурный центр — Визит (связывает клиента, услуги, мастера, оплату)

### Связи
- 1:M: Клиент → Визит
- M:N: Визит ↔ Услуга (через Предоставление услуги)
- M:N: Услуга ↔ Материал (через Расход материалов)
- 1:M: Визит → Оплата
- M:N: Клиент ↔ Услуга (через Отзыв)
- 1:M: Сотрудник → Расписание
- 1:M: Категория услуги → Услуга

### Сложности
- Связь M:N между визитом и услугой — клиент может получить несколько услуг за один визит
- Учёт нормативов расхода материалов на услугу и фактического расхода
- Две оценки в отзыве (услуги и мастера) для детальной обратной связи
- Расписание с типами дней (рабочий/выходной/отпуск)

### Возможности развития системы
- Система лояльности: абонементы на комплекс услуг, бонусные программы, скидки для постоянных клиентов
- Онлайн-запись: интеграция с сайтом салона и мессенджерами, автоматические напоминания о записи клиентам
- Финансовый модуль: расчёт зарплаты мастеров (оклад + %), учёт себестоимости услуг с учётом материалов
- CRM-функционал: история предпочтений клиентов, дни рождения, персональные рекомендации мастеров и услуг
